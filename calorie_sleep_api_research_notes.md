# Calorie Burn and Sleep Quality API Research Notes

## Current narrowed scope

The user clarified that the API search should focus only on companies or platforms that can provide active calories, total calories burned, and possibly sleep quality data. Nutrition, barcode, food logging, restaurant macros, and unrelated integrations should be deferred for now.

## Terra

Official documentation states that Terra Health & Fitness API uses subscription-based usage credits. It includes 100,000 monthly credits in the subscription, with additional usage billed at $0.005 per credit from 100,001 to 1,000,000 monthly credits and $0.003 per credit beyond 1,000,001 credits. Terra’s credit model counts active authentications and events; an active authentication costs 200 credits per month, and the first 400 events per active authentication are free. The pricing page says events include health, activity, sleep, and workout data. The broader Terra site/search result indicates support for activity, sleep, nutrition, heart rate, HRV, VO2max, body metrics, menstrual cycle, and CGM, but the official pricing page does not itself enumerate active calories or total calories.

Useful source: https://docs.tryterra.co/health-and-fitness-api/pricing

## Garmin Health API

Official Garmin Health API documentation lists health metrics including steps, heart rate, sleep, stress, calories, Body Battery, respiration, blood pressure, and more. The page says Garmin provides data in JSON format and includes heart rate, steps, calories, sleep, respiration, body composition, stress, pulse-ox, and epoch summaries for all-day activities. Access requires approval, and commercial use requires a license fee payment.

Useful source: https://developer.garmin.com/gc-developer-program/health-api/

## Apple HealthKit

Official Apple HealthKit documentation describes HealthKit as a central repository for health and fitness data on iPhone and Apple Watch, accessible with user permission. It is a native iOS framework rather than a web API. It is likely a strong path if JIMMI eventually ships an iOS app or native wrapper, but it is not directly callable from the current web app as a simple server-side API.

Useful source: https://developer.apple.com/documentation/healthkit

## Fitbit Web API / Google Health migration

Official Fitbit documentation states that Fitbit Web APIs retrieve user data from Fitbit trackers, smartwatches, scales, and manually entered logs, with user consent. Endpoint groups include Activity and Activity Time Series, Sleep, Heart Rate, HRV, Devices, and Subscriptions. The page also states that the legacy Fitbit Web API will be deprecated in September 2026 and developers must migrate to the Google Health API developer infrastructure.

Useful source: https://dev.fitbit.com/build/reference/web-api/

## Android Health Connect

Official Health Connect documentation says Health Connect stores and structures health and fitness data with user permission. It includes heart rate, step count, and sleep data taken in sessions, and offers APIs for reading, writing, and synchronizing data. It is an Android platform API rather than a web API, so it is a strong path for an Android app/native wrapper but not a direct current web-only integration.

Useful source: https://developer.android.com/health-and-fitness/health-connect

## Oura API

Official Oura API documentation says the Oura API provides access to data generated by the Oura Ring. It uses OAuth2 applications for multiple users, with a default ten-user limit until the application is approved for wider audience release. The extracted entry page did not itself detail calorie or sleep endpoint fields, so follow-up documentation is needed for V2 data types.

Useful source: https://cloud.ouraring.com/docs/

## WHOOP API

Official WHOOP developer documentation confirms OAuth-accessible V2 data models for sleep, cycle, and recovery. The Sleep data model includes `score_state` and a `score` object with `sleep_performance_percentage`, `sleep_consistency_percentage`, `sleep_efficiency_percentage`, respiratory rate, stage summary, and sleep needed. The Cycle data model includes `strain`, `kilojoule`, `average_heart_rate`, and `max_heart_rate`; calories can be derived from kilojoules by dividing by 4.184, but the official Cycle model does not expose a simple `total_calories` field in the viewed documentation. Recovery exposes a 0–100 `recovery_score`, resting heart rate, HRV, SpO2, and skin temperature. WHOOP therefore looks useful for sleep quality and recovery, but less direct than Oura or Fitbit for active calories and total calories.

Useful sources: https://developer.whoop.com/docs/developing/user-data/sleep/ ; https://developer.whoop.com/docs/developing/user-data/cycle/ ; https://developer.whoop.com/docs/developing/user-data/recovery/


## Oura API V2 field confirmation

The extracted Oura V2 OpenAPI documentation confirms `/v2/usercollection/daily_activity` returns `active_calories`, `total_calories`, `target_calories`, `score`, and activity-score `contributors`. It also confirms `/v2/usercollection/daily_sleep` returns a sleep `score` and `contributors`. Oura is therefore the cleanest direct web API match for JIMMI’s narrowed requirements: active calories, total calories, and sleep quality score. The Oura application documentation also states that registered OAuth applications have a default ten-user limit until approved for a wider audience.

Useful sources: https://cloud.ouraring.com/docs/ ; https://cloud.ouraring.com/v2/docs

## Fitbit activity field confirmation

Official Fitbit Daily Activity Summary documentation confirms `summary.activityCalories` as calories burned during active periods above sedentary level, including both activity-burned calories and BMR, and `summary.caloriesOut` as total calories burned for the day. The same endpoint requires the `activity` OAuth scope. This makes Fitbit a strong direct match for active and total calories. Fitbit’s broader Web API documentation confirms Sleep endpoints are available, but Fitbit is transitioning to the Google Health API and legacy Fitbit Web APIs are deprecated in September 2026.

Useful sources: https://dev.fitbit.com/build/reference/web-api/activity/get-daily-activity-summary/ ; https://dev.fitbit.com/build/reference/web-api/

## Google Health API direction

Official Google Health API documentation describes it as the next generation of the Fitbit Web API, covering Fitbit, Pixel Watch, and other sources on Google OAuth 2.0 infrastructure. It is strategically important for any Fitbit integration because legacy Fitbit endpoints are moving to this new infrastructure, but it adds migration uncertainty while the developer surface continues to evolve.

Useful source: https://developers.google.com/health
