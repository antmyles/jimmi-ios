# Voice Lifecycle Diagnosis

The supplied screen recording confirms three behaviors that need correction in the JIMMI Chat voice flow. First, tapping the orb does enter **Listening**, but live speech is displayed in a separate hovering live-transcription bubble rather than being treated as the primary chat-box dictation experience. Second, after the first complete turn, a second tap briefly enters **Listening** and then immediately aborts back to **Tap to speak**, which points to stale browser speech-recognition or recorder cleanup state blocking a new turn. Third, the current client derives visible **Speaking** from both `orbVoiceState === "speaking"` and `jimmiSpeechMutation.isPending`; this can label the experience as Speaking while the app is still preparing audio, even though the orb visual should remain in blue Thinking until actual playback starts.

The code-level diagnosis is that `playJimmiAudio()` currently sets `orbVoiceState` to `speaking` immediately after `audio.load()` and before `audio.play()` has successfully begun. This should be moved into the playback-start success path. The second-turn failure is likely caused by the combination of shared `recognitionRef`, abort/stop flags, and cleanup not fully resetting before a new `MediaRecorder`/`SpeechRecognition` session starts. The patch should add a deterministic cleanup routine before each new capture, clear stale recognition and recorder handlers, keep the UI in **Listening** while live dictation updates the chat composer, then move to **Thinking** only when capture stops and request processing begins.
